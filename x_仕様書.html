通常の投稿フロー（計画投稿）

【事前ヒアリングが必要な投稿のみ・3日前の朝9時】自動通知
選手へのヒアリングが必要なテーマ（note欄が設定されている投稿）の場合だけ、Bot が @担当者 にメンション（sns-botチャンネル）
「📅 3日後の投稿予定：[日付]　テーマ：[テーマ]
　💬 事前ヒアリング：〇〇について各選手に感想を聞いてください。
　そのまま進める場合は「GO」と書いてください。
　ヒアリング結果や変更・追加情報があればそのまま書いてください。」
（例：新キャラ・イングリットの投稿なら「各選手にイングリットの感想を聞いてください」と通知される）

【3日前〜前日】担当者が返信（3パターンから選べる）
　① 「GO」と書く → カレンダー通りのテーマで当日文案を生成
　② 「オニキ：使いやすい。サトル：強い。あしゅまる：対空が面白い」→ 選手コメントをネタに文案生成
　③ 「今日弟がトパチャン勝ったのでそっちで」→ 新ネタで文案生成
　→ 何を書いても必ず3パターンが返ってくる

事前ヒアリングが不要な通常投稿には3日前通知は来ない。当日朝の「前回投稿の完了確認」からそのまま始まる。

【当日・朝8時】前回投稿の完了確認
前回の非時限投稿がまだ「done」になっていない場合、Bot が @担当者 にメンションして確認する（sns-botチャンネル）
「❓ 前回の投稿「[テーマ]」は投稿済みですか？　✅ → 済んでいる　❌ → まだ」
　✅ を押す → 該当投稿を完了扱いにして、そのまま当日文案の生成に進む
　❌ を押す → 前回分を後回しにして今日の分は一旦見送り（!shiftと同じ処理）
　未解決の間はこの確認を毎日繰り返す

【当日・朝8時】自動で文案を送信
前回投稿の確認が済んだら、Bot が sns-bot チャンネルに3パターンの投稿文を送付 ＋ @担当者 メンション

【当日・朝8時・文案送信直後】シフト確認（時限投稿以外）
Botが続けてシフト確認メッセージを送信し、✅ と ❌ のリアクションを付ける：
「↩️ 今日のテーマを別のネタに差し替える場合：✅ → 今日のテーマを次回に回す　❌ → このまま進める」
　✅ を押す → 今日のテーマが次回以降にずれ（1つシフト）、今日は「done」扱いになる
　❌ を押す → そのまま通常通り進む

【当日】担当者が投稿
好きな文案をコピーしてXに貼り付けて投稿

【修正したい場合】sns-botチャンネルに指示を書く
「もっとシンプルに」「サトルのキャラに寄せて」など
→ 必ず3パターンが返ってくる（スマホからでも使える）
→ 気に入ったものをコピーしてXに投稿

Fly.io Volume（schedule.json 永続化）

Fly.io Volume が /data にマウントされている場合、schedule.json は /data/schedule.json に保存され、デプロイをまたいでステータスが保持される。

  • 初回起動時：/data/schedule.json が存在しない場合、バンドル版を自動コピーして初期化する
  • カレンダー更新時：!reload コマンドで最新のバンドル版を取り込む（既存ステータスは保持）
  • Volume未使用時：バンドル版を直接使用（デプロイのたびにステータスがリセットされる従来の動作）

自動タイマーの仕組み

node-cronは使用せず、setInterval（1分毎チェック）で動作する。Fly.ioの共有CPUでcronが実行されないmissed execution問題を回避するため。

時刻（JST）動作条件
8:00〜8:10前回投稿の完了確認 → 当日の投稿文案を自動生成・送信その日まだ送信していない場合のみ1回
9:00〜9:10事前ヒアリングが必要な投稿（note欄あり）のみ3日後の投稿予定を通知その日まだ通知していない場合のみ1回
9:00〜9:10カレンダー残量チェックその日まだ実行していない場合のみ1回
9:00〜9:10Gmail メールチェック → 対象メールをSlackに通知その日まだ実行していない場合のみ1回（ロックファイルで制御）

起動時キャッチアップ

Bot起動時にJSTで8時以降だった場合、当日の未送信文案を自動送信する。JSTで9時以降かつ当日未実行の場合はメールチェックも自動実行する。Fly.ioの再起動後でも朝の処理が届く。

GOコマンドのフォールバック

「GO」と送信した際、Botは以下の順で投稿を特定する：

  1. メモリ上に保持している currentPendingPostId で検索
  2. IDが消えていた場合（再起動後など）、直近の notified または scheduled 状態の投稿を自動検索

再起動後でも「GO」が正しく機能する。

手動コマンド

コマンド動作用途
!draft当日の投稿文案を即時生成・送信8時の自動送信を待たずに今すぐ文案が欲しい時
!notify事前ヒアリング通知（note欄あり）を即時送信9時の自動通知を待たずに確認したい時
!status直近5件の予定と状態を表示カレンダー状況の確認

Gmailメールチェック機能

毎朝9時に contact@northeption.com と info@northeption.com の受信メールを取得し、AIで分類して重要メールのみSlackに通知する。

カテゴリ内容通知
A〜D, G_PRIMEスポンサー・メディア・取材・大会関係など重要メールSlackに通知
I（未分類）AIが判断できなかったメールSlackに通知（要確認）
その他広告・自動送信・無関係メールなどスキップ

通知対象・未分類ともに0件の場合はSlack送信自体がスキップされる（毎日必ず通知が来るわけではない）。

Gmail OAuth認証（トークン更新手順）

トークンが期限切れ（invalid_grant）になった場合、または初回セットアップ時に実行する。

# 1. mail_check-app ディレクトリで再認証
cd ..\mail_check-app
node scripts\auth.js info       # info@northeption.com でブラウザログイン
node scripts\auth.js contact    # contact@northeption.com でブラウザログイン

# 2. 新しいトークンを Fly.io に登録
cd ..\auto_x-app
node setup-secrets.js
注意： Google Cloud ConsoleのOAuth同意画面は「本番」モードで運用すること。テストモードのままだとリフレッシュトークンが7日で失効する。
認証時に「このアプリはGoogleで確認されていません」と出た場合は「詳細」→「移動（安全でないページ）」で進む（社内ツールのため問題なし）。

注意事項

  • DISCORD_BOT_TOKEN と ANTHROPIC_API_KEY はチャットに絶対書かない（Anthropicが自動検出してDiscordがトークンを無効化するため）
  • .env ファイルは絶対にGitにコミットしない（APIキーが漏洩する）
  • Botは必ず1つだけ起動すること。複数起動すると同じ投稿に対して複数の文案が届く
  • X APIは使用していない。投稿は必ず手動でコピペして行う
  • 発信の最終判断は必ず人間が行う（全自動投稿はしない）
  • Fly.io Volume を使用している場合、schedule.json のステータスはデプロイをまたいで保持される。カレンダーを更新した後は !reload を実行すること
  • Fly.io Volume を使用していない場合、schedule.json はDockerイメージに焼き込まれているためデプロイのたびにステータスがリセットされる
