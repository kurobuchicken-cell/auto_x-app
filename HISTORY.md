## 2026-07-13 前回投稿未了(❌)時に元テーマの文案を再送するよう変更

**背景:** 担当者から「当日の3文案が届かない」と問い合わせがあったが、原因は既存の「前回投稿の完了確認」フロー（前回が`done`になるまで当日分を送らない設計、2026-07-08のHISTORYエントリ参照）による正常動作だった。ただし❌（前回は未投稿）を選んだ際、これまでは`executeShift`でスケジュールをスライドするだけで、肝心の未投稿テーマの文案自体は再送されなかった。担当者が文案を紛失・見失っている場合、スライド通知だけでは実際にその場で投稿する手段がない。

**対応:** `messageReactionAdd`の❌ハンドラで、`executeShift`を呼ぶ前に、未投稿だった元投稿の`confirmedContent || theme`を使って`sendDraftsToChannel`を呼び、3文案を再送するようにした。スライド自体の挙動（時限投稿は固定でずらす）は変更していない。

**テストについて:** ローカルの`.env`にDiscordトークンが無く、`index.js`もテスト用に関数を切り出せる構造になっていないため、実機での動作確認は本番反映後に❌リアクションを押して確認する運用とした。

## 2026-07-13 Fly Deployワークフローを削除（大量の失敗通知メールの原因）

**症状:** `.github/workflows/fly-deploy.yml` が残っていたため、mainブランチへのpushのたびに存在しないFly.ioアプリへのデプロイが自動実行され、そのたびに失敗して「Run failed: Fly Deploy」という通知メールが届いていた。本セッションだけでコミットを多数行ったため、同日中に大量の失敗メールが溜まっているとユーザーから指摘があった。

**原因:** `Oracle移行手順書.md`の8章に「監視期間を問題なく終えたらこのワークフローを無効化する」という手順が元々あったが、実施されないまま残っていた。Oracle VM側は既に5日以上安定稼働しており（pm2 uptime確認済み）、監視期間は実質的に終えている状態だった。

**対応:** `gh` CLIがこの環境に無かったため、`gh workflow disable`ではなく`.github/workflows/fly-deploy.yml`自体を削除する形で対応した。GitHub Actions上の過去の実行履歴は残るが、今後のpushではワークフローが起動しなくなる。

**教訓:** 移行作業の「あとで無効化する」系の後片付けタスクは、監視期間終了の判断基準（例：本番の連続稼働日数）を決めておかないと先延ばしになりやすい。今回は「pm2 uptimeが5日超」を実質的な完了判断に使った。

## 2026-07-13 Oracle VMのSSHアクセス喪失からの復旧と、判明した設定ミス

**背景:** MA案件紹介メール機能をOracle VMへデプロイしようとしたところ、ノートPC・家PCどちらにもSSH秘密鍵が無く接続できなかった（`Oracle移行手順書.md`には「SSH接続確認済み」とあるが、鍵の保管場所がどこにも記録されていなかった）。

**復旧方法:** Oracle Cloud ConsoleのRun Command機能・Instance Console Connectionの両方を試したが、Run Commandはエージェントがコマンドを拾わず「Accepted」のまま進行せず、Console ConnectionはUI上に作成導線を発見できなかった（いずれも時間切れで断念）。最終的に、インスタンス作成日（2026-07-07）と同日にダウンロードフォルダへ保存されていた`ssh-key-2026-07-07.key`（OCIがインスタンス作成時に自動生成・ダウンロードさせた鍵）が有効であることが判明し、これで接続できた。

**教訓:** OCIでインスタンス作成時に「鍵を自動生成」を選ぶと、ブラウザのダウンロードフォルダに`ssh-key-<日付>.key`という名前で保存される。この鍵をダウンロードフォルダに置きっぱなしにせず、`~/.ssh/`等の分かりやすい場所に移動し、`dev-config`のsync-list対象に加えておくべきだった。次にOracle VMの鍵を紛失した場合は、まずインスタンス作成日に近い日付のダウンロードフォルダを確認するとよい。

**デプロイ時に判明した設定ミス:**
- プロジェクトCLAUDE.mdに「本番稼働：`pm2 start index.js --name northeption-bot`」と記載されていたが、実際にVM上で稼働しているpm2プロセス名は`northeption-sns-bot`だった（ハイフンの位置違い）。誤った名前で`pm2 restart`すると「Process or Namespace not found」で失敗する。CLAUDE.md側を実際の名前に修正した。
- VM側の`package.json`に、コミット済みの`googleapis: ^140.0.1`とは異なる`^173.0.0`への未コミットの変更が残っていた（過去のトラブル対応で誰かが直接`npm install`した形跡）。`git pull`がこれとコンフリクトしたため、ユーザー確認の上でVM側の変更を破棄し、コミット済みのpackage-lock.jsonで`npm ci`し直した。あわせて、2026-07-08にコミットした「過去のコミット漏れ」分（Dockerfile・README.md・package-lock.json本体など）もこのタイミングまでVMに一度もpullされておらず、まとめて反映された。

## 2026-07-13 CLAUDE.md／.envのPC間同期を「dev-config」リポジトリ経由の自動化に変更

**背景:** これまでノートPC・家PC間のグローバルCLAUDE.mdと各プロジェクトの`.env`は手動コピーで同期しており、コピー忘れによる設定のズレが起きやすかった。加えて、グローバルCLAUDE.md（`~/.claude/CLAUDE.md`）が実は自分のOneDriveへのシンボリックリンクになっていたことが判明（CLAUDE.md本文の「OneDrive経由のシンボリックリンクは使わない」というルールと矛盾した状態で放置されていた）。ユーザーはこれを機にOneDriveの使用自体をやめる方針にした。

**設計判断:**
- OneDriveの代替として、新規プライベートGitHubリポジトリ `dev-config`（https://github.com/kurobuchicken-cell/dev-config）を作成し、CLAUDE.mdと各プロジェクトの`.env`をここに集約管理する方式にした。無料のプライベートリポジトリで完結し、追加費用なし。
- `.env`はセキュリティルール上Claudeが中身を読んではいけないため、同期スクリプト（`sync-push.ps1`/`sync-pull.ps1`）は`Copy-Item`でファイルごとコピーするだけにし、内容を一切読み取らない設計にした。
- 同期のタイミングは「プロジェクトコードのpull」と切り離した。プロジェクトコード（git pull・npm install）は`SESSION_LOG.md`の前回作業環境と比較して環境が変わった場合のみ実行するが、dev-config（CLAUDE.md・.env）はどのプロジェクトを開いていても**毎セッション開始時に無条件でpull**する設計にした。これにより「Aプロジェクトで作業→別PCでBプロジェクトの作業中にCLAUDE.mdを修正→翌日Aプロジェクトに戻る」という経路でも、無条件pullのおかげで最新のCLAUDE.mdが確実に反映される。
- セッション終了時の運用が煩雑にならないよう、「終了」の一言でコミット確認省略・push・`sync-push.ps1`実行・SESSION_LOG.md追記までをまとめて行うトリガーワードを設定した。

**ハマった点:**
- `sync-push.ps1`/`sync-pull.ps1`に最初 `C:\Users\kin\.claude\CLAUDE.md` とパスをハードコードしていたが、家PCのWindowsユーザー名が`kurob`（`kin`ではない）だったため、そのままでは動かない設計ミスだった。`$env:USERPROFILE`ベースに直して両PCで共通に使えるようにした。
- 家PC側にも、ノートPCと同様に`.claude\CLAUDE.md`がシンボリックリンク（おそらく同じくOneDrive由来）として残っていることが判明。上書き配置の前に削除が必要。
- PowerShellの`run_in_background`で起動したプロセスは、直前のコマンドで`$env:Path`を更新していてもそれを引き継がない（プロセスごとに独立）。`gh`コマンドがインストール直後に見つからないというエラーはこれが原因だった。バックグラウンド実行時も毎回`$env:Path`を明示的に再構築する必要がある。
- 家PC側で`sync-pull.ps1`を実行する際、既存の`.env`が中身未確認のまま上書きされる可能性があったため、実行前に`.env.bak`へのバックアップを挟む運用にした（Claude側は中身を読まない方針のため、差異の確認はユーザー側で行う）。

**未実施：**
- 家PC側のセットアップ完了確認（`.env.bak`との内容比較、CLAUDE.md反映確認）はユーザーが別途実施中
- mail関連の未コミット変更（`.env.example`・`HISTORY.md`・`index.js`・`services/mail/`配下）は今回のセッションと無関係のため触れていない。別セッションで対応予定

**実データでの動作確認結果:**

## 2026-07-08 3日前通知の廃止判断と、まとめて通知が届いた件の調査

**症状:** 担当者から「今日一気に投稿済み通知が飛んできた。①なぜまとめて届いたか ②当日は毎回前回投稿確認を聞く仕様ではなかったか」という質問があった。

**調査結果:** `schedule.json`を確認したところ、Fly→Oracle移行作業（7/7）の期間中に投稿確認が止まっていたらしく、過去日付なのに`done`になっていない投稿が8件（うち非時限3件：6/24・6/26・7/1）滞留していた。`sendTodayDrafts`は「直前の非時限投稿が`done`か」を1件ずつ確認し、解消されるたびに再帰的に次を確認する設計のため、滞留分をまとめて解消すると異なる日付・テーマの確認/文案が短時間で連続して届くように見える。前回投稿確認の仕様自体は正常に動作しており、バグではなく滞留の解消が重なった結果と判断した。

**仕様変更の判断:** この調査を機に「3日前の事前通知」機能を担当者に確認したところ、運用してみて不要と判明。ただし選手への事前ヒアリングが必要な投稿（`note`欄がある投稿）だけは、当日いきなり聞かれても選手コメントを集める時間がないため、3日前通知を条件付きで存続することにした。あわせて、3日前通知に同居していた「カレンダー残量チェック（残り30日以下で警告）」は独立した日次チェック（`checkCalendarRemaining`）として切り出した。前回投稿の完了確認フローは今回の変更対象外（既存のまま）。

**対応:** `index.js`の`sendThreeDayNotifications`を廃止し、`sendHearingNotifications`（note欄がある投稿のみ対象）と`checkCalendarRemaining`に分離。`!notify`コマンド・`GO`返信・`currentPendingPostId`によるヒアリング確定フローは、対象をnote欄ありの投稿に限定した上で存続。`仕様書.md`・`システム概要書.html`・`x_仕様書.html`も実装に合わせて更新。

**教訓:**
- 「前回投稿の完了確認」のような都度確認フローは、確認待ち状態をメモリ上だけで持つと、未解決分が積み重なった際に一気に解消されて「まとめて届いた」ように見える。滞留を早期に検知・解消する運用（`!status`でのこまめな確認等）が重要。
- 「3日前に必ず通知する」という一律の設計は、実際に運用してみて初めて「本当に必要なのは事前ヒアリングが要るケースだけ」と分かった。全件一律の自動化ではなく、必要なケースだけに絞る形に落ち着いた。

## 2026-07-07 Fly→Oracle移行でGmailトークンをそのまま移すとinvalid_grantが再発

**症状:** /data永続化データをFlyから移行した際、Fly側でエクスポートした
token_info.json・token_contact.jsonを配置したところ、再起動後のメール
チェックで両アカウントとも`invalid_grant`エラーが発生した。

**根本原因:** Oracle側では移行作業中、/data導入前の時点で既に独自に
（.envのGMAIL_INFO_TOKEN/GMAIL_CONTACT_TOKENから復元した）ローカル版の
トークンファイルで複数回メールチェックを実行しており、その過程でGoogleの
リフレッシュトークンローテーションにより新しいリフレッシュトークンに
更新されていた。その後からFly側のスナップショット（Jun23・当日0時時点）を
上書き配置したため、Google側では既に無効化された古いリフレッシュトークンを
使うことになり、invalid_grantになった。

**解決:** Fly由来のトークンではなく、Oracle上で当日実際に動作実績のある
ローカル版トークン（~/apps/northeption-sns-bot/tokens/配下）を/dataに
採用した。以降はgmail-auth.jsの'tokens'イベントが自動で最新版を
/data配下に書き戻すため、追加対応は不要。

**教訓:** Gmail等リフレッシュトークンローテーションを行うOAuth連携では、
移行元（Fly）のトークンスナップショットが必ずしも最新とは限らない。
移行先で並行してAPIを叩いた実績がある場合は、その場で実際に成功して

静的データとは異なり、トークン類は「最後に成功した側」を正とする。

## 2026-07-07 Oracle移行作業中にFly.io側の二重稼働が発覚・/data永続化データを移行

**症状:** Oracle VM側でメール通知のデバッグ作業を進めている最中に、Fly.io側の
`/data`ボリュームを確認したところ、本日7:00 JST時点のrun_lock・sent_ids.json・
token_info.jsonが更新されており、**Fly側のBotも並行して稼働し続けていた**ことが
判明した。移行手順書では「本番切替まではFlyを止めない」設計だったが、実際に
Oracle側で動作確認を進める段階になっても明示的にFlyを止めておらず、結果として
同一Discordトークン・同一メールアカウントに対して両環境が同時に処理を行う状態に
なっていた（メールのSlack通知が二重送信された可能性がある）。

**対応:**
1. `flyctl scale count 0`でFly側マシンを停止（アプリ・ボリュームは保持）
2. データ取得のため一時的に`scale count 1`で1分程度だけ再起動し、
   `schedule.json`・`tokens/token_info.json`・`tokens/token_contact.json`・
   `logs/sent_ids.json`を取得後、即座に`scale count 0`へ戻した
   （本日分のrun_lockが既に存在したため、再起動してもメール二重送信は
   発生しなかった）
3. Oracle側に`/data`ディレクトリを新規作成し取得したデータを配置。これにより
   コード内の`fs.existsSync('/data')`分岐でFlyと同じ永続化パスを使うようになり、
   スケジュール状態・Gmailトークン・通知済みIDの継続性が確保された
4. 作業中に`.env.save`という追跡外ファイルが`.gitignore`で除外されていない
   ことも発覚したため、`.gitignore`に`.env.*`・`/logs/`・`/tokens/`を追加

**教訓:**
- 「本番切替までは並行稼働させない」という設計は、Fly側を明示的に止める
  タイミングを手順として決めておかないと、テスト段階でなし崩し的に両方稼働する
  状態になりやすい。次回以降は移行作業に着手する時点でまずFlyを止める運用にする。
- `fly ssh sftp`はマシンが起動していないと使えない（`scale count 0`後は
  ボリュームへアクセスできなくなる）。データ取得は停止前に済ませるか、
  取得のためだけに一時的に再起動する必要がある。
- Windows Git Bashから`flyctl ssh sftp get /data/...`を実行する際、MSYS的な
  パス変換で`/data`が`C:/Program Files/Git/data`等に化けるため、
  `MSYS_NO_PATHCONV=1`を付ける必要がある。

## 2026-07-07 Oracle移行時にSLACK_WEBHOOK_URLの設定ミスでSlack通知が失敗

**症状:** Oracle VMへ移行後、メールチェック機能はGmail取得・分類まで正常に動作したが、
Slack通知で`TypeError: Invalid URL`が発生していた（input: undefined → その後の調査で
プレースホルダー文字列 → 最終的にSlack User IDの値）。

**根本原因:** `.env`への手動追記時に`SLACK_WEBHOOK_URL`と`SLACK_MENTION_USER_ID`の値を
取り違えており、最終的に両方ともUser IDの値になっていた（正しいWebhook URLが未設定）。
加えて、`.env`編集後にPM2プロセスを再起動していなかったため、そもそも新しい値が
反映されていなかった（dotenvはプロセス起動時にしか`.env`を読まない）。

**診断の落とし穴:** `/proc/<pid>/environ`はexec時点のOS環境のスナップショットであり、
dotenvが実行時に`process.env`へ書き込んだ値は反映されない。稼働中プロセスの環境変数
確認にこれを使うと「値が読み込まれていない」と誤診断する（今回実際に誤診断した）。
実際にはdotenvは正常にパースできていた。

**解決:**
1. 当日分の`run_lock`ファイルを削除
2. PM2を再起動（`.env`再読込のため必須）
3. 正しい`SLACK_WEBHOOK_URL`の値は、ユーザーのローカル端末からssh標準入力経由で
   VMへ直接転送し、VM上ではファイル操作のみで`.env`の該当行を置換（Claudeは値を
   一切参照しない方式）
4. 起動時キャッチアップでメールチェック→Slack送信まで再テストし成功を確認

**教訓:**
- `.env`を編集しただけでは稼働中プロセスには反映されない。編集後は必ずPM2再起動が必要。
- Webhook URLやAPIキー等の機密値をAIエージェント経由で転記させると誤入力・混同の
  リスクがある。ユーザー自身の端末からssh標準入力経由で直接転送する方式にすることで、
  値をチャット履歴やコマンド引数に残さず、AIが値を参照せずに安全に移行できる。

## 2026-06-23 前回投稿の完了確認機能を追加（再発防止）

**背景:** 上記の滞留が起きた根本原因は、Botが「文案を送った」ことと「実際にXへ投稿された」ことを区別できていなかったこと。`sendTodayDrafts`は文案送信に成功すると`status: 'drafts_sent'`にするだけで、その後実際に投稿されたかを確認する仕組みがなかった。

**対応:** `sendTodayDrafts`実行時、直前の非時限投稿（時限はスキップして遡る）が`done`になっていなければ、本日分を送る前に「前回の投稿は済んでいるか」をリアクション（✅/❌）で確認するようにした。
- ✅ → 該当投稿を`done`化し、本日分を通常送信
- ❌ → 時限を除いてスケジュールを1つスライドし、本日分はスキップ。未解決の前回投稿は解決するまで毎日再質問する

機能投入にあわせて、過去の確認済み投稿4件（6/1・6/5・6/6・6/8）を`schedule.json`上で`done`化した。これをしないと初回実行時にすでに済んでいる古い投稿を再確認してしまうため。

**教訓:** 「文案を送った」と「投稿が完了した」は別の状態として管理する必要がある。Bot側の自動判定と人間の確認応答を明確に分けることで、滞留を早期発見できるようにした。
