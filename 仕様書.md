# NORTHEPTION 自動通知システム 仕様書

最終更新：2026-07-08

---

## 1. システム概要

NORTHEPTIONの日常業務における2種類の通知を自動化するシステム。

| 通知 | トリガー | 通知先 |
|------|----------|--------|
| X投稿文案の生成・スケジュール管理 | 毎日8〜9時（JST） | Discord |
| Gmailの重要メール仕分け | 毎日9時（JST） | Slack |

- 動作環境：Fly.io（東京リージョン）、1マシン・1プロセス
- エントリーポイント：`index.js`（Node.js 18+）

---

## 2. 動作スケジュール

毎分ポーリング（`setInterval` 60秒）でJST時刻をチェックし、該当時間帯に1日1回実行。

| 時刻（JST） | 処理 | 出力先 |
|------------|------|--------|
| 8:00〜8:09 | 当日の投稿テーマでX文案3パターンを生成（前回投稿の完了確認を含む） | Discord `#sns-bot` |
| 9:00〜9:09 | 事前ヒアリングが必要な投稿（note欄あり）のみ3日前通知 | Discord `#sns-bot` |
| 9:00〜9:09 | 投稿カレンダー残量チェック（残り30日以下で警告） | Discord `#sns-bot` |
| 9:00〜9:09 | 前日（月曜は土〜日）のGmailを分類してレポート | Slack |

起動時キャッチアップ：8時以降に起動した場合は当日文案を、9時以降は当日メールチェックを即時実行する。

## 3. 通知①：X投稿文案生成（Discord）

### 概要

`schedule.json` に登録された投稿スケジュールに沿って、当日の投稿テーマをClaude APIに渡し、X（Twitter）投稿文案3パターンを生成してDiscordに送信する。担当者はDiscordからコピーしてXに手動投稿する。

### 処理フロー

```
[8:00 JST]
↓ schedule.json から当日分（status ≠ drafts_sent/done）を取得
↓ confirmedContent があればそれを、なければ theme を使用
↓ Claude API（claude-opus-4-7）に3パターン生成を依頼
↓ Discord に1パターンずつ3件に分けて送信（スマホコピー対応）
↓ timeSensitive でなければ「今日のテーマを差し替えますか？」ボタン（✅/❌）を表示
↓ schedule.json の status を drafts_sent に更新
```

### 前回投稿確認フロー（当日文案生成の直前に実行）

```
[8:00 JST]
↓ 直近の非時限投稿で status ≠ done のものがあるか確認
↓ あれば「前回の投稿は済んでいますか？」をDiscordに送信（✅/❌）し、当日分の生成はここで一旦停止
↓ ✅ → 該当投稿を done にして当日文案生成へ進む
↓ ❌ → 非時限投稿だけを1つ後ろ倒し（!shiftと同じ処理）、当日分はスキップ
↓ 未解決の間は毎日この確認を繰り返す
```

### 事前ヒアリング通知フロー（note欄がある投稿のみ）

```
[9:00 JST]
↓ schedule.json から 今日+3日（status = scheduled かつ note が設定されている）投稿を取得
↓ テーマ・事前ヒアリング内容をDiscordに通知
↓ status を notified に更新、currentPendingPostId に記録
↓ 担当者が「GO」→ そのままテーマ確定
↓ 担当者が別テキスト（ヒアリング結果など）を返信 → confirmedContent として保存
↓ どちらの場合もその場で3パターンの文案を返す
```

note欄がない投稿は3日前通知を送らず、当日の前回投稿確認フローのみで処理される。

### 投稿スケジュール管理（schedule.json）

Fly.io Volume（`/data/schedule.json`）で永続化。

**投稿オブジェクト**

```json
{
  "id": "2026-06-10",
  "date": "2026-06-10",
  "theme": "投稿テーマ",
  "status": "scheduled",
  "confirmedContent": null,
  "note": "事前ヒアリング内容（省略可）",
  "timeSensitive": true
}
```

**status の遷移**

```
scheduled → notified → confirmed → drafts_sent → done
```

note欄がない投稿は `notified` / `confirmed` を経由せず `scheduled → drafts_sent → done` で完結する。

**timeSensitive フラグ**：`true` の投稿は `!shift` によるずらし対象から除外（大会結果報告・誕生日など）。

**カレンダー残量チェック**：毎日9時に単独で実行。残り30日以下になるとDiscordへ警告を送信。

### Discordコマンド

`#sns-bot`（ID: `1507312995041280070`）のみ受け付け。

| コマンド | 動作 |
|----------|------|
| `!draft` | 当日文案生成を手動実行 |
| `!notify` | 事前ヒアリング通知（note欄あり）を手動実行 |
| `!status` | 直近5件の予定と待機中IDを表示 |
| `!shift` | 今日のテーマを次回以降に後ろ送り（timeSensitive除外） |
| `!reload` | Volume上のschedule.jsonをリポジトリ版で上書き更新（ステータス保持） |
| `GO`（返信） | 直近の通知済み投稿をテーマそのままで確定 |
| その他テキスト | 即時で3パターンの文案を生成して返す |

### Claude API（文案生成）

- モデル：`claude-opus-4-7`
- システムプロンプトに選手情報・投稿スタイル・大会スケジュール・禁止事項を定義
- プロンプトキャッシュ（`cache_control: ephemeral`）でコスト削減
- 出力形式：`【パターン1：シンプル】` / `【パターン2：熱量高め】` / `【パターン3：ファン巻き込み】`

## 4. 通知②：Gmailメールレポート（Slack）

### 概要

`info@northeption.com` と `contact@northeption.com` の2アカウントを毎日9時にチェックし、重要メールを分類してSlackに日次レポートとして送信する。

### 処理フロー

```
[9:00 JST]
↓ Gmail API で前日分のメールを2アカウント分取得
↓ 送信済みIDを除外（重複通知防止）
↓ ルールベースで分類（ドメイン→キーワードの優先順）
↓ ルールで分類できなかったメールをClaude APIで推測分類
↓ 通知対象（A/B/C/D/G'）と未分類（I）があればSlackに送信
↓ 送信済みIDを /data/logs/sent_ids.json に記録
↓ 90日超のログを自動削除
```

### 対象期間

| 曜日 | 対象期間 |
|------|----------|
| 月曜 | 土曜〜日曜（2日分） |
| 火〜日 | 前日（1日分） |

### メール分類カテゴリ

| カテゴリ | 内容 | Slack通知 |
|----------|------|-----------|
| 🔴 A | 請求書・決済 | ✅ |
| 🟠 B | 契約・法務 | ✅ |
| 🟢 C | 採用・応募 | ✅ |
| 🔵 D | 案件・ビジネス提案 | ✅ |
| 🟡 G' | 決済通知（MoneyForward） | ✅ |
| ⚪ I | 未分類（要確認） | ✅ |
| ⬜ E | ゲームPR・インディー | スキップ |
| ⬜ F | SNS通知 | スキップ |
| ⬜ G | SaaS通知 | スキップ |
| ⬜ H | 営業・メルマガ | スキップ |

### 分類ロジック（優先順）

1. `config.json` の skipDomains → H扱い
2. `config.json` の notifyDomains → 指定カテゴリ
3. `config.json` の notifyKeywords → 指定カテゴリ
4. 組み込みドメインルール（shopify.com → A、cloudsign.jp → A/B、moneyforward.com → G'/G など）
5. 組み込みキーワードルール（件名マッチ）
6. 上記すべてに該当しない → Claude API（`claude-sonnet-4-6`）で推測し、I（未分類）として通知

### ルールの動的追加

`services/mail/config.json` を編集するだけでルールを追加できる（デプロイ不要）。

```json
{
  "skipDomains": ["example-pr.com"],
  "notifyDomains": { "partner.com": "D_案件・ビジネス提案" },
  "notifyKeywords": { "スポンサー契約": "B_契約・法務" }
}
```

### Slackメッセージ形式

```
📬 メール日次レポート｜2026/06/10(水)　対象: 6/9(火)
━━━━━━━━━━━━━━━━━━
🔴 請求書・決済 (1件)
• [INFO] Shopify — 利用明細 📎

🟢 採用・応募 (2件)
• [CONTACT] 田中太郎 — タレントマネージャー募集の件
• [CONTACT] 株式会社XX — グラフィックデザイナー応募の件

━━━━━━━━━━━━━━━━━━
@ユーザー 通知対象: 3件 ／ 未分類: 0件 ／ 受信総数: 12件 ／ スキップ: 9件
```

- 通知対象・未分類がともに0件の場合はSlackに送信しない
- Slack送信失敗時は最大3回リトライ

## 6. データ永続化

Fly.io Volume（`schedule_data`、マウント先：`/data`）を3用途で共有。

| パス | 内容 | 管理主体 |
|------|------|----------|
| `/data/schedule.json` | X投稿スケジュール | sns-bot |
| `/data/tokens/` | Gmail OAuthトークンファイル | mail-check |
| `/data/logs/sent_ids.json` | 送信済みメールID（90日保持） | mail-check |
| `/data/logs/run_log.json` | メールチェック実行ログ（最新200件） | mail-check |
| `/data/logs/classification_YYYY-MM-DD.json` | 日別分類ログ（90日保持） | mail-check |

## 8. 再認証（OAuthトークン更新）が必要な場合

Gmail OAuthトークンの有効期限が切れた場合は、ローカルの `mail_check-app` で再認証する。

```
# info@northeption.com
npm run auth:info

# contact@northeption.com
npm run auth:contact
```

生成されたトークンを `GMAIL_INFO_TOKEN` / `GMAIL_CONTACT_TOKEN` として Fly.io に再設定する。
