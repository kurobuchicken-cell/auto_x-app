仕様書 — X投稿文案管理 ＋ メール日次レポート

作成：2026年6月10日　動作環境：Fly.io 東京リージョン

1. システム概要

NORTHEPTIONの日常業務における2種類の通知を自動化するシステム。1つのサーバー（Fly.io）上で1プロセスとして稼働する。

通知トリガー処理内容通知先
X投稿文案の生成・スケジュール管理 毎日8〜9時（JST） 投稿カレンダーに基づき文案3パターンを自動生成。 Discord
Gmailの重要メール仕分け 毎日9時（JST） 2アカウントのメールを自動分類してレポート送信。 Slack
項目内容
動作環境Fly.io 東京リージョン（nrt）、256MB・shared CPU、24時間稼働
アプリ名northeption-sns-bot
言語Node.js 18+
主要ライブラリdiscord.js v14、@anthropic-ai/sdk、Google Gmail API

2. 動作スケジュール

毎分ポーリング（setInterval 60秒）でJST時刻をチェックし、該当時間帯に1日1回実行する。node-cronは使用しない（Fly.io共有CPUでのmissed execution対策）。

時刻（JST）処理通知先
8:00〜8:09当日の投稿テーマでX文案3パターンを生成・送信（前回投稿の完了確認を含む）Discord #sns-bot
9:00〜9:09事前ヒアリングが必要な投稿（note欄あり）のみ3日前通知Discord #sns-bot
9:00〜9:09投稿カレンダー残量チェック（残り30日以下で警告）Discord #sns-bot
9:00〜9:09前日分（月曜は土〜日）のGmailを分類してレポート送信Slack
起動時キャッチアップ：8時以降に起動した場合は当日文案を、9時以降はメールチェックを即時実行する。Fly.ioの再起動後でも通知が漏れない。

3. 通知① X投稿文案生成

Discord #sns-bot

3-1. 概要

schedule.json に登録された投稿スケジュールに基づき、当日の投稿テーマをClaude APIに渡してX投稿文案3パターンを生成し、Discordに送信する。担当者はDiscordからコピーしてXに手動投稿する。X APIは使用しない。最終判断は必ず担当者が行う。

3-2. 前回投稿確認フロー（毎日8時・当日文案生成の直前）

1
直近の非時限投稿で status ≠ done のものがあるか確認
↓
2
あれば「前回の投稿は済んでいますか？」をDiscordに送信（✅/❌）。当日分の生成はここで一旦停止。
↓
3a
✅ → 該当投稿を done にして当日文案生成へ進む
3b
❌ → 非時限投稿だけを1つ後ろ倒し（!shiftと同じ処理）、当日分はスキップ
↓
4
未解決の間は毎日この確認を繰り返す

3-3. 事前ヒアリング通知フロー（毎日9時・note欄がある投稿のみ）

1
schedule.json から「今日+3日」かつ status = scheduled かつ note が設定されている投稿を取得
↓
2
テーマ・事前ヒアリング内容をDiscordに通知。status を notified に更新。
↓
3
担当者が「GO」と返信 → テーマそのままで confirmedContent に保存
担当者が別テキストを返信 → その内容を confirmedContent に保存
↓
4
どちらの場合もその場で文案3パターンを返す
note欄がない投稿は3日前通知を送らず、当日の前回投稿確認フローのみで処理される。

3-4. 当日文案生成フロー（毎日8時）

1
schedule.json から当日分（status ≠ drafts_sent / done）を取得
↓
2
confirmedContent があればそれを使用、なければ theme を使用
↓
3
Claude API（claude-opus-4-7）に3パターン生成を依頼
↓
4
Discord に1パターンずつ3件に分けて送信（スマホコピー対応）
↓
5
timeSensitive でなければシフト確認ボタン（✅/❌）を表示
↓
6
schedule.json の status を drafts_sent に更新
担当者が何を書いても必ず3パターンが返る。1対3を厳守。

3-5. 投稿スケジュール管理（schedule.json）

Fly.io Volume（/data/schedule.json）で永続化。

投稿オブジェクトの構造

{
  "id": "2026-06-10",
  "date": "2026-06-10",
  "theme": "投稿テーマ",
  "status": "scheduled",
  "confirmedContent": null,
  "note": "事前ヒアリング内容（省略可）",
  "timeSensitive": true
}

status の遷移

scheduled → notified → confirmed → drafts_sent → done

note欄がない投稿は notified / confirmed を経由せず scheduled → drafts_sent → done で完結する。

timeSensitive フラグ：true の投稿は !shift によるずらし対象から除外される（大会結果報告・誕生日など）。

カレンダー残量チェック：毎日9時に単独で実行。残り30日以下になるとDiscordに警告を自動送信する。

3-6. Discordコマンド一覧

チャンネル #sns-bot（ID: 1507312995041280070）のみ受け付ける。

コマンド動作
!draft当日文案生成を手動実行
!notify事前ヒアリング通知（note欄あり）を手動実行
!status直近5件の予定と現在の待機IDを表示
!shift今日のテーマを次回以降に後ろ送り（timeSensitive除外）
!reloadVolumeのscheduleをDockerイメージ最新版で上書き更新（既存ステータス保持）
GO（返信）直近の通知済み投稿をテーマそのままで確定
その他テキスト即時で文案3パターンを生成して返す

3-7. Claude API（文案生成）

項目内容
モデルclaude-opus-4-7
max_tokens1024
プロンプトキャッシュcache_control: ephemeral（コスト削減）
出力形式【パターン1：シンプル】 / 【パターン2：熱量高め】 / 【パターン3：ファン巻き込み】

システムプロンプトに選手情報（5名・絵文字・Twitterハンドル）、投稿スタイル（#NthWIN #SF6必須・黄色いハート禁止・140字以内）、大会スケジュール・誕生日を定義済み。

4. 通知② Gmailメールレポート

Slack

4-1. 概要

info@northeption.com と contact@northeption.com の2アカウントを毎日9時にチェックし、重要メールを分類してSlackに日次レポートとして送信する。通知対象・未分類がともに0件の場合はSlackに送信しない。

4-2. 処理フロー（毎日9時）

1
Gmail API で対象期間のメールを2アカウント分取得
↓
2
送信済みID（/data/logs/sent_ids.json）と照合して重複を除外
↓
3
ルールベースで分類（ドメイン → キーワードの優先順）
↓
4
ルールで分類できなかったメールをClaude API（claude-sonnet-4-6）で推測分類
↓
5
通知対象（A/B/C/D/G'）または未分類（I）があればSlackにレポート送信
↓
6
送信済みIDを記録・90日超のログを自動削除

4-3. 対象期間

曜日対象期間
月曜土曜〜日曜（2日分）
火〜日前日（1日分）

4-4. メール分類カテゴリ

カテゴリ内容Slack通知
🔴 A：請求書・決済請求書、利用明細、Shopify決済など✅ 通知
🟠 B：契約・法務契約書、覚書、GMOサインなど✅ 通知
🟢 C：採用・応募求人応募、エントリー、採用関連✅ 通知
🔵 D：案件・ビジネス提案スポンサー、取材、協賛、提案✅ 通知
🟡 G'：決済通知（MF）MoneyForwardの決済完了通知✅ 通知
⚪ I：未分類（要確認）ルールで分類できなかったメール。Claude推測ラベルを付与。✅ 通知
⬜ E：ゲームPR・インディーゲームPR会社からのメールスキップ
⬜ F：SNS通知Instagram、Facebookなどの通知メールスキップ
⬜ G：SaaS通知各種SaaSサービスからの自動通知スキップ
⬜ H：営業・メルマガ営業メール、メルマガ、配信停止案内スキップ

4-5. 分類ロジック（優先順）

優先順判定内容
1config.json の skipDomains に含まれるドメイン → H扱い（スキップ）
2config.json の notifyDomains に含まれるドメイン → 指定カテゴリ
3config.json の notifyKeywords に含まれるキーワード（件名マッチ） → 指定カテゴリ
4組み込みドメインルール（shopify.com → A、cloudsign.jp → A/B、moneyforward.com → G'/G など）
5組み込みキーワードルール（件名に「請求書」「合意締結」「スポンサー」など）
6上記すべて該当なし → Claude API で推測し、I（未分類）として通知

4-6. ルールの動的追加

services/mail/config.json を編集するだけでルールを追加・変更できる（デプロイ不要）。

{
  "skipDomains": ["example-pr.com"],
  "notifyDomains": { "partner.com": "D_案件・ビジネス提案" },
  "notifyKeywords": { "スポンサー契約": "B_契約・法務" }
}

6. データ永続化（Fly.io Volume）

Fly.io Volume（volume名: schedule_data、マウント先: /data）を3用途で共有する。

パス内容
/data/schedule.jsonX投稿スケジュール（ステータス管理）
/data/tokens/Gmail OAuthトークンファイル
/data/logs/sent_ids.json送信済みメールID（90日保持）
/data/logs/run_log.jsonメールチェック実行ログ（最新200件）
/data/logs/classification_YYYY-MM-DD.json日別分類ログ（90日保持・自動削除）
Volume未使用時はリポジトリ同梱の schedule.json をフォールバックとして使用する（デプロイのたびにステータスがリセットされる）。

8. 運用メモ・注意事項

セキュリティ
  • APIキー・トークンはチャットに絶対書かない（Anthropicが自動検出してキーが無効化される）
  • .env ファイルはGitにコミットしない
  • Botは必ず1つだけ起動すること。複数起動すると重複通知が発生する

カレンダー更新手順

# schedule.json を編集・コミット後にデプロイ
flyctl deploy --remote-only

# デプロイ完了後、VolumeのJSONをイメージ版で更新
# → Discord #sns-bot に送信
!reload

Gmail OAuthトークンの再認証が必要になった場合

トークンの有効期限が切れた場合は、ローカルの mail_check-app で再認証する。

# info@northeption.com
npm run auth:info

# contact@northeption.com
npm run auth:contact

生成されたトークンを GMAIL_INFO_TOKEN / GMAIL_CONTACT_TOKEN として Fly.io に再設定する。
